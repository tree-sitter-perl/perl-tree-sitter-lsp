# Task: Framework Accessor Synthesis (Moo/Moose, Mojo::Base, DBIC)

**Branch:** create `framework-accessors` from `main` (after inheritance merges)
**Prerequisite:** inheritance chain resolution must be merged first — this builds on `package_parents` and `resolve_method_in_ancestors`

## Goal

When a user types `$obj->` on a Moo/Moose/Mojo::Base object, show the accessors generated by `has` declarations. When they type `$row->` on a DBIC result class, show column accessors and relationship methods. Goto-def, hover, and return type resolution all work for these synthesized methods.

Currently, `has 'name' => (...)` in a Moo class produces zero symbols — the builder sees it as a plain function call and ignores it. This task adds framework-aware accessor synthesis during the builder's single-pass CST walk.

## Design principle

**Synthesize symbols, not framework models.** The builder recognizes DSL calls and emits standard `SymbolDetail::Sub` entries with appropriate params and return types. Everything downstream — completion, hover, goto-def, inheritance — works automatically because it already operates on symbols.

This is the same pattern used for Perl 5.38 `field $x :reader` (builder.rs line 803): the builder sees the field declaration and synthesizes a `SymKind::Method` symbol. We extend this to framework DSL calls.

## Framework detection

The builder already tracks `use` statements. Add a per-file `framework_mode` flag set during `visit_use_statement`:

```rust
#[derive(Debug, Clone, Copy, PartialEq)]
enum FrameworkMode {
    None,
    Moo,       // use Moo; or use Moo::Role;
    Moose,     // use Moose; or use Moose::Role;
    MojoBase,  // use Mojo::Base 'Parent'; or use Mojo::Base -base;
}
```

Detection rules:
| `use` statement | Mode | Side effect |
|---|---|---|
| `use Moo` | `Moo` | — |
| `use Moo::Role` | `Moo` | — |
| `use Moose` | `Moose` | — |
| `use Moose::Role` | `Moose` | — |
| `use Mojo::Base 'Parent'` | `MojoBase` | Also add `Parent` to `package_parents` |
| `use Mojo::Base -base` | `MojoBase` | — |
| `use Mojo::Base -strict` | `None` | No accessors (just imports strict) |

**Scope:** `framework_mode` is per-package, not per-file. A file with `package Foo; use Moo;` and `package Bar;` should only synthesize Moo accessors for `Foo`. Store as `HashMap<String, FrameworkMode>` keyed by package name, or reset on `package` statement if single-package files dominate.

**Mojo::Base inheritance:** `use Mojo::Base 'SomeParent'` is both a framework declaration AND an inheritance declaration (it does `push @ISA, 'SomeParent'` at import time). The builder should write to `package_parents` here, just like `use parent`. The import list extraction already handles the `'SomeParent'` argument.

## Moo/Moose `has` synthesis

### CST pattern

`has` is called as a function: `has name => (key => val, ...)`. In tree-sitter-perl this is an `ambiguous_function_call_expression` (no parens on `has`) with:
- `function` field: bareword `has`
- arguments: either a `list_expression` containing the name + fat-comma pairs, or for single-arg `has 'name'`, a direct `string_literal`

Multi-attribute: `has [qw(foo bar)] => (is => 'ro')` — the first argument is a `quoted_word_list` inside array brackets.

### What to extract

From each `has` call, extract:
1. **Attribute name(s)** — string literal, qw list, or arrayref of strings
2. **`is` value** — `'ro'`, `'rw'`, `'lazy'`, `'bare'` (or absent)
3. **`isa` value** — type constraint string (optional)
4. **`handles` value** — delegation (future scope, skip for now)

### Accessor synthesis rules

| `is` value | Accessor | Params | Notes |
|---|---|---|---|
| `'ro'` | `name()` | none | Read-only getter |
| `'rw'` | `name()` / `name($val)` | `($val)` optional | Getter/setter |
| `'lazy'` | `name()` | none | Read-only (built on first access) |
| `'bare'` | — | — | No accessor generated |
| absent | `name()` | none | Moo: `'bare'` default. Moose: `'bare'` default. Most real code specifies `is`. |
| `'rwp'` (Moo only) | `name()` + `_set_name($val)` | | Reader + private writer |

When `is` is absent, Moo and Moose technically don't generate an accessor. But in practice, 99% of `has` declarations include `is`. For the absent case, **don't synthesize** — avoids false positives.

### Return type mapping from `isa`

Map common type constraint strings to `InferredType`:

| `isa` value | `InferredType` | Notes |
|---|---|---|
| `'Str'` | `String` | |
| `'Int'`, `'Num'` | `Numeric` | |
| `'HashRef'` | `HashRef` | |
| `'ArrayRef'` | `ArrayRef` | |
| `'CodeRef'` | `CodeRef` | |
| `'RegexpRef'` | `Regexp` | |
| `'Bool'` | `Numeric` | Perl booleans are 0/1 |
| `'ClassName'` (Moose) | — | Too vague |
| `InstanceOf['Foo']` (Moo) | `ClassName("Foo")` | Parse the bracket content |
| `'Foo::Bar'` (Moose) | `ClassName("Foo::Bar")` | Moose allows class names as types |

**Parsing `isa` from the CST:** The `isa` value is a string literal in a fat-comma pair. Extract it as text. For `InstanceOf['Foo']`, regex/string parse the bracket content. For Moose bare class names, check if the string contains `::` or starts with uppercase — heuristic, not perfect.

If `isa` is a coderef (`isa => sub { ... }`) or a Type::Tiny expression without string representation, skip — return type stays `None`.

### Symbol synthesis

For each extracted accessor, call `self.add_symbol(...)` exactly like the existing `:reader`/`:writer` pattern:

```rust
self.add_symbol(
    attr_name.clone(),           // e.g. "name"
    SymKind::Method,
    node_to_span(has_call_node), // span of the whole `has` call
    attr_name_span,              // selection span on the attribute name string
    SymbolDetail::Sub {
        params: if is_rw { vec![ParamInfo { name: "$val".into(), default: None, is_slurpy: false }] } else { vec![] },
        is_method: true,
        return_type: mapped_isa_type,  // from isa mapping above, or None
        doc: None,
    },
);
```

The `selection_span` should point to the attribute name in the `has` call (the string literal), so goto-def lands on the `has` declaration. This matches how `:reader` points to the `field` declaration.

### Extracting fat-comma pairs from arguments

The builder already has `extract_use_import_list` for parsing argument lists. For `has` we need a different helper that extracts key-value pairs from a fat-comma list:

```rust
fn extract_fat_comma_pairs(&self, node: Node) -> Vec<(String, String)>
```

Walk the `list_expression` children. Fat commas (`=>`) pair adjacent bareword/string keys with their values. Extract value as text for string literals, or identify node kind for coderefs/arrayrefs.

**This is a general-purpose helper** — it'll be useful for DBIC too. Keep it generic.

### `has` with `+` prefix (attribute modification)

`has '+name' => (default => 'foo')` modifies an inherited attribute. This is common in Moose subclasses. **Skip for Phase 1** — the accessor already exists from the parent. If we synthesize a new one, we'd create a duplicate. The inheritance chain already surfaces the parent's accessor.

## Mojo::Base `has` synthesis

### Key difference: fluent setters

Mojo::Base accessors return `$self` when called as setters, enabling chaining:
```perl
$obj->name('Alice')->age(30)->save;
```

This means the setter return type is `ClassName(current_package)`, not `None`.

### CST pattern

Same as Moo — `has` is an `ambiguous_function_call_expression`. But Mojo::Base `has` is simpler:
- `has 'name'` — no options hash
- `has 'name' => 'default'` — default value (string/number)
- `has 'name' => sub { {} }` — default value (coderef)
- `has [qw(foo bar)]` — multiple attributes
- `has [qw(foo bar)] => 'default'` — multiple with shared default

No `is`/`isa` options. All accessors are read-write.

### Synthesis rules

Every `has` produces a read-write accessor. Return type:
- **Getter** (no args): `None` (unknown what it holds), unless we can infer from the default value
- **Setter** (with args): `ClassName(current_package)` — fluent return

Since we can't distinguish getter vs setter at the call site in the CST, **use the fluent return type** (`ClassName(current_package)`). This is correct for chaining (`$obj->name('x')->age(5)`) and acceptable for getter usage (the hover will show the class type, which is technically what the variable holds if chained).

Actually, more precisely: store the return type as `ClassName(current_package)`. This enables chaining resolution. For simple getter use like `my $name = $obj->name`, the variable gets typed as the class — imperfect but not harmful (it won't trigger wrong completions since Perl is duck-typed).

### Default value type inference

If the default is a literal, we can infer:
- `has 'items' => sub { [] }` → accessor could return `ArrayRef`
- `has 'config' => sub { {} }` → accessor could return `HashRef`

**Skip for Phase 1.** Default inference is a nice-to-have. The fluent return type is more valuable.

## DBIC Result Class synthesis

### Detection

DBIC result classes are identified by:
```perl
use base 'DBIx::Class::Core';
# or
use parent -norequire, 'DBIx::Class::Core';
# or
__PACKAGE__->load_components('Core');  # older style
```

The builder already handles `use base`/`use parent` for inheritance. Add a DBIC flag when the parent chain includes any `DBIx::Class` package:

```rust
fn is_dbic_class(&self) -> bool {
    if let Some(ref pkg) = self.current_package {
        if let Some(parents) = self.package_parents.get(pkg) {
            return parents.iter().any(|p| p.starts_with("DBIx::Class"));
        }
    }
    false
}
```

This covers the common case. `load_components` is harder (runtime method call) — detect it as `__PACKAGE__->load_components(...)` where any argument contains `"Core"` or `"ResultSource"`. Heuristic, not exhaustive.

### `add_columns` / `add_column`

```perl
__PACKAGE__->add_columns(
    id    => { data_type => 'integer', is_auto_increment => 1 },
    name  => { data_type => 'varchar', size => 255 },
    email => { data_type => 'varchar' },
);
```

Or the simpler form:
```perl
__PACKAGE__->add_columns(qw(id name email));
```

Each column name becomes a read-write accessor method.

**CST pattern:** `method_call_expression` with `invocant` = `__PACKAGE__` (or package name), `method` = `add_columns`/`add_column`. Arguments are a fat-comma list where every other key (the odd-positioned entries) are column names, or a `qw()` list.

**Synthesis:** For each column name, emit:
```rust
SymbolDetail::Sub {
    params: vec![ParamInfo { name: "$val".into(), ... }],
    is_method: true,
    return_type: None,  // column value type unknown
    doc: None,
}
```

### Relationship accessors

```perl
__PACKAGE__->has_many(comments => 'Schema::Result::Comment', 'post_id');
__PACKAGE__->belongs_to(author => 'Schema::Result::User', 'author_id');
__PACKAGE__->has_one(profile => 'Schema::Result::Profile', 'user_id');
__PACKAGE__->might_have(avatar => 'Schema::Result::Avatar', 'user_id');
__PACKAGE__->many_to_many(tags => 'comment_tags', 'tag');
```

| Method | Accessor name | Return type |
|---|---|---|
| `has_many` | 1st arg | `ClassName("DBIx::Class::ResultSet")` (or the related class if we want to be more precise — but ResultSet is what you actually get) |
| `belongs_to` | 1st arg | `ClassName(2nd_arg)` — returns the related row object |
| `has_one` | 1st arg | `ClassName(2nd_arg)` — returns the related row object |
| `might_have` | 1st arg | `ClassName(2nd_arg)` — returns the related row object (or undef) |
| `many_to_many` | 1st arg | `ClassName("DBIx::Class::ResultSet")` |

**CST pattern:** Same `method_call_expression` with `__PACKAGE__->relationship_method(...)`. First argument is the accessor name (string/bareword), second is the related class (string).

**Synthesis:** Emit method symbol with appropriate return type. The return type enables chaining like `$user->comments->search(...)` — if we type `comments()` as returning a ResultSet, and ResultSet is in the module index with its methods, we get completion on the result.

### `set_primary_key`, `table`, `resultset_class`

These are configuration, not accessors. **Skip** — don't synthesize anything.

### `resultset_attributes`, `add_unique_constraint`

Not accessors. **Skip.**

## Implementation plan

### Cross-file propagation

The synthesized methods are standard symbols. The subprocess export extraction (`subprocess_main`) already runs the full builder — so a resolved Moo module will have its `has`-generated methods in the `subs` map automatically. Cross-file completion, hover, and goto-def work without any additional changes.

**One thing to verify:** The export extraction only includes methods that are in `@EXPORT` or `@EXPORT_OK`. Moo accessors aren't exported — they're methods on the class. But `complete_methods_for_class` + inheritance resolution already walks the module index's `subs` map regardless of export status. So cross-file `$obj->accessor` will work as long as the module's `subs` map includes the accessor. **Check:** Does the subprocess emit ALL subs, or only exported ones? If only exported, we need to also emit methods.

Actually, looking at `subprocess_main` — it only iterates `export.iter().chain(export_ok.iter())` for the subs map. This means Moo accessors (which aren't in `@EXPORT`/`@EXPORT_OK`) won't appear in cross-file `subs`. **This needs fixing** — the subprocess should also emit all methods (symbols with `is_method: true`) regardless of export status. This is already a gap for regular OOP methods on non-Moo classes too.

### Subprocess method emission fix

In `subprocess_main` and `resolve_and_parse`, after the export loop, add:

```rust
// Also emit all methods (for cross-file OOP resolution)
for sym in &analysis.symbols {
    if sym.is_method && !subs.contains_key(&sym.name) {
        // emit to subs map
    }
}
```

This is orthogonal to the framework work but required for cross-file method resolution to work for ANY class, not just Moo. **Should be a separate commit/PR but is a blocker for cross-file framework accessor resolution.**

## Bundled: Inferred param types in `ExportedSub`

This version requires a schema bump anyway (framework synthesized methods in the cache). Bundle this improvement at the same time.

### Problem

Cross-file signature help shows param names but NOT inferred types. Locally, `signature_for_call` calls `analysis.inferred_type(&param.name, sig_info.body_end)` to discover that e.g. `$timeout` is `Numeric` (from `$timeout + 1` usage in the body). For cross-file subs, `body_end` is `Point::new(0, 0)` — a nonsensical position — so the lookup returns None.

The data exists: the subprocess runs the full builder on each module, and the resulting `FileAnalysis` has TypeConstraints for params based on body usage. We just don't serialize it.

### Fix

**1. Add `inferred_type` to `ExportedParam`:**

```rust
pub struct ExportedParam {
    pub name: String,
    pub is_slurpy: bool,
    pub inferred_type: Option<String>,  // NEW: type tag ("Numeric", "String", "Object:Foo", etc.)
}
```

**2. Subprocess extraction** (`module_resolver.rs`):

When building `ExportedParam` from a `ParamInfo`, also query the type:

```rust
// For each param, look up its inferred type at the end of the sub body
let param_type = analysis.inferred_type(&param.name, sym.span.end)
    .map(|t| inferred_type_to_tag(t));
ExportedParam {
    name: param.name.clone(),
    is_slurpy: param.is_slurpy,
    inferred_type: param_type,
}
```

**3. JSON serialization:** Add `"type": "Numeric"` to each param object in the subprocess JSON. Only include when present.

**4. SQLite:** Already stored as JSON in the `subs` column — the new field is just another key per param object. Schema bump forces cache invalidation.

**5. Signature help** (`file_analysis.rs`):

In `signature_for_call`'s `CrossFile` branch, map `ExportedParam.inferred_type` into the `ParamInfo`:

```rust
ResolvedSub::CrossFile { params: exported_params, .. } => {
    let params: Vec<ParamInfo> = exported_params.iter().map(|p| ParamInfo {
        name: p.name.clone(),
        default: None,
        is_slurpy: p.is_slurpy,
    }).collect();
    // ...
}
```

And in `symbols.rs` `signature_help`, the CrossFile path formats param types directly from the `ExportedParam` instead of calling `inferred_type` with `body_end`:

```rust
// For cross-file params, use the carried type instead of querying body_end
let type_label = match &sig_info.param_types {
    Some(types) => types.get(i).and_then(|t| t.as_ref()),
    None => analysis.inferred_type(&p.name, sig_info.body_end),
};
```

Alternatively, simplify: add `inferred_type: Option<String>` to `ParamInfo` itself, populated from `ExportedParam` for cross-file and from `analysis.inferred_type()` for local. Then `signature_help` just reads `p.inferred_type` uniformly. Cleaner.

### Tests

```rust
#[test]
fn test_exported_param_carries_inferred_type() {
    // Subprocess-style: build FA, extract ExportedParam with type
    let fa = build_fa("
        sub process {
            my ($self, $count, $name) = @_;
            my $x = $count + 1;      # $count is Numeric
            my $y = $name . 'suffix'; # $name is String
        }
    ");
    let ty = fa.inferred_type("$count", Point::new(4, 0));
    assert_eq!(ty, Some(&InferredType::Numeric));
}
```

## What this does NOT cover

- `handles` delegation (`has 'http' => (handles => [qw(get post)]`) — synthesizes delegated methods. Future scope.
- Moose `MooseX::*` extensions — infinite rabbit hole. Skip.
- `around`/`before`/`after` method modifiers — these wrap existing methods, don't create new ones. No synthesis needed.
- DBIC `load_components` beyond simple string matching — runtime metaprogramming, can't fully resolve.
- DBIC `ResultSet` custom methods — user writes these as normal subs, already handled.
- Moo/Moose `BUILD`/`BUILDARGS` — these are regular subs the user writes. Already handled.
- `MooX::StrictConstructor` / `Moose::Util::TypeConstraints` — extensions. Skip.
