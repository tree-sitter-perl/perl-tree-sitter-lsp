## Summary

Rows: 35 total — **24 PASS, 2 FAIL, 9 REVIEW**.

By repo: Exporter-Tiny 6 (3 PASS / 1 FAIL / 2 REVIEW + ET-8 REVIEW = 3 PASS / 1 FAIL / 3 REVIEW), Sub-Exporter 5 (4 PASS / 1 REVIEW), URI 8 (7 PASS / 1 REVIEW), DateTime 7 (7 PASS), Log-Log4perl 7 (5 PASS / 1 FAIL / 1 REVIEW).

FAIL/REVIEW patterns worth a human's attention:

- **FAIL — bareword call refs dropped when the call is an rvalue of `my $x = f(...)`** (ET-4): `mkopt`'s `@EXPORT_OK`/def/`@{ mkopt($in) }` refs are found, but the three `my $opts = mkopt(\@args)` call sites are missing. The `@{ mkopt(...) }` form IS found, so the gap correlates with the bareword call being the direct RHS of a `my` declaration.
- **FAIL — chained method on an inferred return resolves only when the return type is inferrable** (L4P-3): `get_logger`'s return type is None (dispatch returns a `Log::Log4perl::Logger`), so `$logger->level` dead-ends. DateTime's chains work because its constructors carry a `return_arms`-inferred `DateTime` type.
- **REVIEW — cross-file def resolves but hover/references for the same imported token does not** (ET-8 hover on imported `fib`, L4P-7 hover on FQ `to_level`): goto-def lands correctly yet hover returns "No hover info" on the identical cursor. Two independent repos, same shape — likely one missing hover→cross-file path.
- **REVIEW — import-list member / generated sub navigation** (ET-3 lands on a sibling call site instead of the def; ET-7 generated `embiggen` lands on an unrelated `use` line).
- **REVIEW — `use_ok`-imported binding untracked** (SE-1): Test::More `use_ok("Mod", qw(fn))` is not treated as an import, so `fn` calls don't resolve. Common test idiom.
- **REVIEW (not a bug) — scheme-dispatched subclass method** (URI-6): `URI->new` types as `URI`; `port` lives in descendant `URI::_server`, unreachable by static ancestor walk. Expected limitation, recorded for completeness.
